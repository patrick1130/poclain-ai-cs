# 微信AI客服系统改进计划

## 9.3 技术债务与改进方向详细规划 (Poclain 工业级架构版)

### 9.3.1 底层架构与代码演进

#### 当前架构现状与瓶颈

* **状态**：核心模块（如 `message_handler.py`）已完成工业级重构，具备防注入沙盒与限流能力。数据库已从 `lazy="dynamic"` 迁移至 `selectin` 急加载，解决了 N+1 性能陷阱。
* **技术债务**：
  1. 数据库驱动仍使用同步的 `pymysql`，在 FastAPI 异步事件循环中执行大批量读写时存在短暂阻塞风险。
  2. RAG 参数（Threshold、Top K）目前已开发前端 UI，但在后端尚未完全实现基于 Redis 的内存级热重载。
  3. 缺乏自动化的 PDF 深度解析流水线（目前依赖基础切分，对复杂工业图表和多栏排版支持有限）。

#### 改进措施


| **优化项**           | **具体措施**                                                                                                 | **优先级** | **预计工作量** |
| -------------------- | ------------------------------------------------------------------------------------------------------------ | ---------- | -------------- |
| **全异步 IO 重构**   | 引入`aiomysql`和`AsyncSession`，替换现有同步 SQLAlchemy 引擎，彻底释放高并发吞吐能力。                       | 高         | 3人天          |
| **动态参数热更新**   | 部署 Redis 缓存层，打通前端 UI，实现 RAG 匹配阈值、召回数量的毫秒级热加载，降低对数据库的 IO 压力。          | 中         | 2人天          |
| **文档预处理流水线** | 开发专门的知识库清洗脚本，引入 OCR 与基于语义的智能分块（Semantic Chunking）技术，优化 PDF 读取精度。        | 高         | 5人天          |
| **微信异步主动投递** | 彻底解决微信被动回复 5 秒超时限制。调整为先回传空字符串挂起连接，AI 生成完毕后调用“客服消息 API”主动推送。 | 高         | 3人天          |

### 9.3.2 深度测试覆盖与安全攻防

#### 当前现状与瓶颈

* **状态**：系统已具备物理级的安全防线（SecurityGuardian）和模型级的认知防线（Temperature 0.0 + System Prompt）。
* **技术债务**：缺乏针对大模型特性的自动化安全回归测试，以及针对高并发场景的极限压测。

#### 改进措施


| **测试项**              | **具体措施**                                                                                                   | **优先级** | **预计工作量** |
| ----------------------- | -------------------------------------------------------------------------------------------------------------- | ---------- | -------------- |
| **Prompt 注入攻防测试** | 建立自动化红蓝对抗（Red Teaming）脚本，定期向系统发送数十种变种“越权、扮演、欺诈”指令，验证拦截率。          | 高         | 4人天          |
| **RAG 幻觉率评估**      | 构建 Poclain 专属测试集（Ground Truth），自动化评估 AI 回答与官方手册的事实一致性，确保幻觉率严格为 0。        | 高         | 5人天          |
| **并发熔断压测**        | 使用 Locust 模拟展会等极端场景下 500+ 用户同时扫码并发咨询，验证限流器（Rate Limiter）与数据库连接池的稳定性。 | 中         | 3人天          |
| **端到端链路测试**      | 覆盖“微信发信 -> 安全拦截 -> Rerank 提纯 -> LLM 生成 -> 微信下发”的完整链路自动化测试。                      | 中         | 4人天          |

### 9.3.3 检索性能与 AI 算力优化

#### 当前现状与瓶颈

* **状态**：已引入阿里 `gte-rerank` 交叉重排模型，极大提升了检索精度（Top 15 -> Top 3）。
* **技术债务**：Rerank 过程增加了额外的网络请求开销；对于高频出现的通用技术问题（如“MS05参数”），每次都重复调用大模型会导致 API 成本浪费。

#### 改进措施


| **优化项**                    | **具体措施**                                                                                                   | **优先级** | **预计工作量** |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------- | ---------- | -------------- |
| **语义缓存 (Semantic Cache)** | 引入 Redis 或 GPTCache，对于相似度 > 95% 的历史高频问题，直接返回缓存答案，实现 O(1) 级毫秒响应与零 API 成本。 | 高         | 4人天          |
| **向量检索降维加速**          | 优化 ChromaDB 索引结构，在知识库规模突破 10 万片段后，无缝平滑迁移至企业级向量库（如 Milvus）。                | 低         | 6人天          |
| **私有化模型预研**            | 考虑到 Poclain 核心技术资料的安全合规性，预研基于 vLLM 本地部署 Qwen-14B 的方案，彻底切断公网数据流出。        | 中         | 10人天         |

## 实施计划蓝图

### 第一阶段（首月）- 生产跃迁与合规落地

* 推进域名 ICP 备案与公安联网备案。
* 完成 Docker 生产环境打包、Nginx 代理与 HTTPS 部署。
* 实现微信 5 秒超时异步主动下发改造。
* 上线前端动态参数调整与 Redis 缓存联动。

### 第二阶段（第 2-3 个月）- 高并发与架构深潜

* 全面完成 `aiomysql` 的异步数据库重构。
* 引入 Semantic Cache（语义缓存）拦截 40% 的高频废算力请求。
* 完善自动化红蓝对抗与防注入渗透测试。
* 开发专门的 PDF 智能切分与 OCR 导入流水线。

### 第三阶段（第 4-6 个月）- 业务闭环与多模态

* 上线“商机自动捕获与流转”引擎（对接企业微信）。
* 引入图文回复能力，支持展示产品结构图。
* 探索图片识别查型号（多模态识别）预研。

## 预期收益评估

1. **绝对的合规与安全性**：通过多重物理隔离与 0.0 温度锁，实现 100% 的公关安全与零幻觉工业答复。
2. **极低的边际成本**：结合 Reranker 精排与语义缓存，大幅降低无效的大模型 Token 消耗。
3. **高吞吐抗压能力**：异步架构与 Redis 将使单台 4C8G 服务器的并发承载能力从 50 QPS 提升至 500+ QPS。
4. **业务价值的直接变现**：不仅是客服工具，更是 24 小时在线的技术型商机“捕获网”，直接反哺销售业绩。
